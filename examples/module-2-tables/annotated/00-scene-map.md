# Scene map — Module 2: Tables

The Tables module is the densest in the set — three cuts share one repo:

- **V1 — Tables Intro** (concept: what a table *is*): 6 scenes.
- **V2 — Using Tables in Workflows** (the orientation cut, `module-2-tables-v2`):
  the **7 scenes that carry the distinctive pattern** — table as queue + record,
  cell-fill write-back, run-twice no-op. **These are annotated in full** (files
  `01`–`07`); they're the reason to read this example.
- **V3 — Workflow Columns / Active Tables** (the pipeline comes to the data):
  5 scenes.
- Plus a **legacy V2 take** kept for the rejection it documents (the floating
  output chips — see PATTERN.md and case 18).

Below: the full-depth seven, then one line each for the rest — intent + the
reused mechanic. The transferable move lives in PATTERN.md and the `01`–`07`
files; everything here either builds toward it or is a sibling concept cut.

---

## The pattern-carrying seven (full annotations: `01`–`07`)

1. **the-table** — push through the workspace `Table` tile into the `leads`
   grid; it assembles, category empty, status `unprocessed` (the **queue**).
   *Mechanic: zoom-through transition + chrome-then-values assemble; the one set
   piece established.*
2. **rows-and-fields** — a row selects (record), releases, then the empty
   `category` column selects (field). *Mechanic: the one-lit-range-=-one-thing
   selection grammar that the run reuses.*
3. **the-workflow** — camera eases to identity; the docs' read→classify→write
   chain assembles below. *Mechanic: camera-over-fixed-geometry reveal; block-
   then-edge assemble.*
4. **the-read** — the run fires: Query ring on, five rows light as one range in
   sync, `<table.rows>` glows then resolves to `[5 rows]`; held live through the
   cut. *Mechanic: cause→effect by synchrony (no connector line); freeze-cut.*
5. **the-write** — same run completes: Update goes live, written range lights,
   values **dip-swap into the cells**, status flips to `qualified`; run releases
   but **values persist**. *Mechanic: cell-fill write-back via writeMix + DipSwap
   + green `cellTint` output residue — never a floating chip.*
6. **the-record** — world dims, run inspector rises with the query record (logs +
   output tree, `rows` highlighted). *Mechanic: record-panel overlay; the read's
   pre-write snapshot.*
7. **queue-and-record** — query fires again, status header glints, **nothing
   lights** (all `qualified`); camera eases back to the final hold. *Mechanic: the
   run-twice no-op — the table remembers.*

## V1 — Tables Intro (concept cut: what a table is)

- **SystemsOverListsScene** — cold open: look-alike records stack in one by one,
  framing "a list of records" before the table noun. *Reused: record-stack
  build-in (a precursor to row-as-record).*
- **OneOffVsManyScene** — two sketches: one input→agent→output (dims) vs. many
  records→process each→write back (wins focus). *Reused: dim-the-loser focus
  contrast; the read→process→write loop in miniature — the V2 run's seed.*
- **TableAnatomyScene** — keystone definition: an empty grid materializes; rows =
  records, columns = fields, type-chip row reveals. *Reused: the SimTable
  assemble + the record/field axes that scene 2 of the V2 cut re-teaches.*
- **WhatBelongsScene** — cells fill with what tables hold; each column highlights
  as it's named (status · score · category · output). *Reused: per-column
  highlight-in-turn (colHeaderHighlight) and the cell-fill content function.*
- **WorkVsInspectableScene** — workflows do the work (left), tables keep it
  inspectable (right); an arrow ties them. *Reused: two-panel work-vs-record
  framing — the conceptual root of "queue and record."*
- **PipelineComesToDataScene** — Video-3 teaser: the workflow icon slides ONTO
  the table and a column lights. *Reused: object-onto-surface motion; sets up V3.*

## V3 — Workflow Columns / Active Tables (the pipeline comes to the data)

- **OldPipelineScene** — the old way: row-wise work as disconnected hand-glued
  parts; broken connectors + warning glints show coordination overhead visually.
  *Reused: state-via-visuals (broken edges = friction, no text label).*
- **WorkflowColumnScene** — keystone V3 beat: a column header becomes a workflow
  column, then runs once per row top-to-bottom, filling each output cell.
  *Reused: the cell-fill write-back, now driven per-row by the column itself.*
- **InputOutputColumnsScene** — made explicit (a spike found it confusing): left
  columns are INPUT (blue tint), right are OUTPUT (green tint). *Reused: the
  `cellTint` input/output color language straight from SimTable.*
- **CascadeScene** — three workflow columns (enrich→qualify→outreach): col 1
  fills, unblocks col 2, unblocks col 3 — a dependency wave. *Reused: staggered
  per-column fill as a dependency cascade.*
- **TableIsTheAgentScene** — final framing: the whole table pulses as one active
  control surface, all workflow columns glowing together. *Reused: whole-table
  highlight as "active"; the V3 thesis image.*

## Legacy V2 take (kept for the rejection — see PATTERN.md / case 18)

- **TheTableScene** — V2 opener (support-tickets table). *Superseded by
  the-table.*
- **ReadRowsScene** — a Table block reads one row; a data pill rides a connector
  line into the block. *The connector-line read the V2 cut replaced with
  synchrony.*
- **ProcessRowScene** — agent emits a structured-output card (category/urgency/
  confidence). *A separate floated card — superseded.*
- **WriteBackScene** — **the rejected one**: output chips ("billing"/"high"/
  "done") fly from the agent into the table on a connector. *"these look
  disgusting" — replaced by cell-fill write-back. The reason the move exists.*
- **TableAsDashboardScene** — pull back: the filled table as a dashboard, each
  output column glowing in turn. *Reused: per-column glow recap.*
