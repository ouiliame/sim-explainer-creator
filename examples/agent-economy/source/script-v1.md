# The Agent Economy — deploy as MCP, agents calling agents (v1) — feat/agent-economy

Registered as `agent-economy-v1` (new video; nothing overwritten).
Self-contained ~68s animated piece; settings-page walkthroughs (MCP
Servers / MCP Tools modals) are screen-rec intercut territory, not
rebuilt here.

**The one idea:** deploy a workflow as an MCP tool and it becomes
infrastructure other agents call; the same protocol lets your agent call
theirs — agents composing in both directions.

**Macro arc:** capability-first — the working agent runs normally (orient
from module-5/6 knowledge), then ONE change (deploy onto an MCP server)
and everything after is that change behaving: one stranger's call, then
five clients at once (the money shot), then the flip (your agent consumes
theirs), then both directions in a single closing run. No manufactured
problem.

**Run count: 9** — (1) editor run, scene 1: baseline "runs when you run
it"; (2) Claude Desktop call, scene 3: the thesis demonstrated — an
outside AGENT's call is just a run; (3–6) the rush, scene 4: four quick
calls (Cursor, Claude Code, VS Code, Sim), one per client badge — the
MULTIPLICITY is this beat's lesson, each run differing from the previous
in exactly one dimension (its caller), played at compressed cadence with
overlapping in-flight pulses; (7) outbound call, scene 5: the flip — the
run that proves your agent consumes their tool (chip rings, pulse out and
back); (8) inbound+outbound bookend run, scene 6: one traversal carrying
both directions; (9) is NOT a run — the final frame holds. Runs 3–6 are
the deliberate, justified exception to minimum-runs: a single call cannot
show "many strangers depend on this"; the count is the content.

**Grounding:** every value traces to `demo-corpus/grounding-v1.md`
(product sources for URL shape / client list / tool naming / colors /
glyphs; invented Scout corpus for companies and briefs). Real server UUID
and run-record numbers: ⟨pending⟩, off screen.

## Motion language

Values resolve in rows (`DipSwap` / `ResolvedTag`); wires carry `WirePulse`
/ `PathPulse` light only; state via product language (rings / 0.35 dim /
green ok). Video-specific additions, each meaning-bearing:

- **Client badges** — house-grammar mini-cards (neutral card, solid brand
  chip + white ported mark, short name) for the product's own MCP client
  list: Cursor, Claude Code, Claude Desktop, VS Code, Sim. They ARE the
  off-canvas callers made visible at the canvas edge. Badge state speaks
  the product language: blue ring while its call is in flight, green ring
  as its reply lands, 0.35 dim when not focal.
- **Caller spokes** — smooth-step edges from each badge to the entry's
  target handle (drawn on when the badge joins). Pulses ride them inbound;
  reply pulses ride them back outbound. The out-stub to the right edge is
  retired in favor of replies returning to their caller — in MCP the
  response goes back to the calling client, not "out the other side."
- **Tool pill** — the module-6 address pill carrying the MCP identity:
  green live dot + `research_competitor · sim.ai/api/mcp/serve/…` (mono).
  Appears at deploy, persists to the end.
- **Partner spoke** — one smooth-step edge from the Agent block's header
  up-right to the partner server badge (`pricing_intel`, MCP purple chip,
  product McpIcon). Outbound tool-call pulses ride it out and back.

## Locked scene list (~68s visual minimum)

1. **an-agent-you-built** (~9s) — [assemble + run]
   The Scout chain (Start → Agent "Scout" → Response) assembles on the
   builder canvas: blocks stagger in, edges draw on; the Agent carries its
   real rows (Model `claude-sonnet-4-6`, Messages `Research <start.input>`)
   and two tool chips (Search, Docs). One editor run: `Vantra Labs`
   resolves into Start's Input row, start blip, pulse, Agent live ring +
   `<start.input>` resolves, Search chip rings briefly (it worked), pulse,
   `<scout.content>` resolves to `"brief ready · 4 sources"` inside the
   Response Data template. Hold, revert.
   *Beat intent: you built a real agent — and it runs when you run it.*

2. **deploy-as-a-tool** (~10s) — [morph-swap]
   Downstream dims to 0.35; the entry is focal. With the editing ring on,
   the entry header crossfades Start → API (deploy; the module-6 morph,
   reprised fast) while the Input row stays pixel-identical. Then the MCP
   pill fades in above the entry: green live dot, `research_competitor ·
   sim.ai/api/mcp/serve/…` — the workflow now has a tool name on a server
   URL. World undims. Hold on the deployed state.
   *Beat intent: deploying onto an MCP server gives the same chain a tool
   name and an address other agents can discover.*

3. **a-stranger-calls** (~11s) — [run]
   A client badge — Claude Desktop — pops in at the left edge (middle
   slot); its spoke draws on to the entry handle. Its ring goes blue and a
   pulse rides the spoke in: `Northwind AI` resolves into the Input row
   and the chain runs exactly as scene 1 (live ring, tag resolutions,
   `"brief ready · 4 sources"`). A reply pulse rides the spoke BACK and
   the badge flashes green. Rows revert; the badge and spoke STAY.
   *Beat intent: an MCP client's tool call is just a workflow run — an
   agent you've never met now drives your agent.*

4. **the-rush** (~15s — the money shot) — [run ×4, multi-caller]
   Four more badges pop in around Claude Desktop — Cursor, Claude Code,
   VS Code, Sim (the product's exact client list, product order top to
   bottom) — each spoke drawing on as it lands. Then the calls come:
   Cursor → Claude Code → VS Code → Sim, each a compressed run (badge
   blue + pulse in + input dip-swap + live ring + resolve + reply pulse
   back + badge green flash), the cadence tightening so late pulses are
   in flight while earlier replies are still returning. Distinct inputs
   per run (`Helio Robotics`, `Quartzline`, `Parcelio`, `Octave
   Systems`). After the last reply, every badge settles; rows revert.
   *Beat intent: you published once — now five different agents, on five
   different platforms, treat your workflow as infrastructure.*

5. **you-call-theirs** (~12s) — [morph-swap (gift) + run]
   The frame's focus flips: badges and entry dim to 0.35; the Agent block
   is focal. A partner server badge pops at the TOP-RIGHT (`pricing_intel`,
   MCP-purple chip with the product's MCP glyph — another team's deployed
   workflow) and a spoke draws from the Agent's header up to it. The
   Agent's Tools row grows a third chip, `pricing_intel`, at exact natural
   width (the product's MCP chip: #6366F1 + MCP glyph). Then one run:
   `Octave Systems` resolves in (editor-origin), Agent live, the NEW chip
   rings while a pulse rides the partner spoke OUT and BACK — their agent
   running somewhere else — then `<scout.content>` resolves. Revert rows;
   chip, badge, spoke stay; world undims.
   *Beat intent: the protocol is symmetric — someone else's deployed
   agent is now a tool inside yours.*

6. **the-agent-economy** (~11s) — [settle / bookend]
   Camera eases back ~7% over the unchanged layout: five client badges
   left, the chain center, the partner server top-right, the tool pill
   above the entry. One final run carries both directions: Claude Code's
   badge lights and its pulse rides in (`Drift Harbor`), the chain walks
   green in causal order, and MID-RUN the `pricing_intel` chip rings as a
   pulse goes out to the partner and returns — then the reply rides back
   to Claude Code, green. Hold the balanced frame.
   *Beat intent: every agent is both a tool and a tool-user — that's the
   agent economy, and your workflow is already in it.*

## Continuity contract

- ONE set piece: the Scout chain at fixed `layout.ts` geometry
  (blockX/CHAIN_Y), with badges, spokes, and pill at layout-owned slots.
  Scenes differ only in state props on `EconomyRig`. Nothing relayouts.
- Entry morph keeps height EXACTLY constant (single Input row, header
  crossfade only). The Tools-row chip gift in scene 5 grows at exact
  natural width/height (SimBlock reveal machinery); scene 6 enters with
  the chip at reveal 1.
- Boundaries (exit == enter, verified by pixel diff):
  - 1→2: plain chain, entry=Start, no pill/badges/spokes, rows template.
  - 2→3: deployed state A — entry=API, pill on, no badges.
  - 3→4: deployed state B — + Claude Desktop badge (slot 3) + its spoke,
    rows template.
  - 4→5: deployed state C — all five badges + spokes, rows template.
  - 5→6: full ecosystem — + partner badge, partner spoke, `pricing_intel`
    chip at reveal 1, world undimmed, rows template.
- All resolutions revert before their scene ends; no freeze-cuts. Scene 6
  ends eased-back and green (no following boundary).
- No sentences on screen; short noun labels (badge names, tool names,
  the pill address) only; narration carries every word.

## Batch-mode assumptions (the converted gates)

1. **Beats gate (Phase 2):** capability-first over problem-first; the
   module-6 deploy morph is reprised rather than re-taught (audience has
   module 6 or doesn't need it — the pill carries the new information).
   Swap cost: low (scene 2 is self-contained).
2. **Script gate (Phase 3):** the run-economy exception in scene 4 (four
   quick runs) is taken deliberately — the director's mandate names this
   exact money shot ("calls arriving from multiple off-canvas clients at
   once, each client badge lighting as its run lands"). Swap cost: medium
   (cutting to 2 runs shortens scene 4 by ~6s).
3. **Replies return to callers** (no right out-stub for inbound calls) —
   product-true for MCP (the result goes back to the client). Deviates
   from module-6 v2's out-stub language. Swap cost: low.
4. **Badge brand chips:** Anthropic clay `#D97757` and VS Code `#007ACC`
   are not in the product CSS (the product renders these clients as text
   buttons); declared in grounding. Swap cost: trivial (constants).
5. **No record-panel scene** (no real artifact, batch rule: invented run
   numbers are forbidden). Swap cost: additive later.
6. **Hype register** is carried by pacing (scene-4 cadence, overlapping
   pulses) and narration energy, NOT by new motion primitives or
   non-product visuals.
